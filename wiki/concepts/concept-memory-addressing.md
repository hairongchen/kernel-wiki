# Memory Addressing

Memory addressing on x86 involves a three-stage translation pipeline from the address a program uses to the electrical signals on the memory bus. Linux uses this hardware machinery in a specific way — minimizing segmentation complexity while relying heavily on paging — to create a portable, flat virtual address space model.

## Address Types

| Address type | Definition |
|-------------|------------|
| **Logical address** | The address generated by a machine instruction, consisting of a **segment selector** (16-bit) and an **offset** (32-bit). This is what the CPU instruction operand specifies. |
| **Linear address** (virtual address) | The result of applying segmentation to a logical address. A 32-bit (or 64-bit) value in a flat, contiguous address space. |
| **Physical address** | The actual address on the memory bus, produced by applying paging to the linear address. 32 bits on standard x86, up to 36 bits with PAE, 52 bits on x86-64. |

The translation pipeline is:

```
Logical address  --[segmentation]--> Linear address  --[paging]--> Physical address
```

## Segmentation

The x86 **segmentation unit** translates logical addresses to linear addresses using segment descriptors stored in the **Global Descriptor Table (GDT)** and **Local Descriptor Table (LDT)**.

Each segment descriptor defines:
- **Base address** — added to the offset to produce the linear address
- **Limit** — maximum valid offset
- **Type/permissions** — code, data, system; read, write, execute
- **DPL** (Descriptor Privilege Level) — ring 0-3

### Linux's Flat Memory Model

Linux effectively **neutralizes** segmentation by setting all segment bases to 0 and limits to 0xFFFFFFFF (4 GB). This means:

```
linear address = 0 + offset = offset = logical address
```

Linux defines four main segments:

| Segment | Selector | Base | Limit | DPL | Purpose |
|---------|----------|------|-------|-----|---------|
| `__KERNEL_CS` | 0x60 | 0 | 0xFFFFFFFF | 0 | Kernel code |
| `__KERNEL_DS` | 0x68 | 0 | 0xFFFFFFFF | 0 | Kernel data |
| `__USER_CS` | 0x73 | 0 | 0xFFFFFFFF | 3 | User code |
| `__USER_DS` | 0x7B | 0 | 0xFFFFFFFF | 3 | User data |

The only practical role of segmentation in Linux is **privilege-level enforcement**: user-mode code (ring 3) cannot access kernel-mode segments (ring 0). Address translation is performed entirely by the paging unit.

A few special-purpose segments remain:
- **TSS segments**: one per CPU, for hardware task-switching support (the kernel uses only the `esp0` field for kernel stack pointer on privilege transitions)
- **Per-CPU data segments**: used on SMP for efficient per-CPU variable access
- **TLS segments**: three per thread, set via `set_thread_area()` for thread-local storage (used by glibc's `%gs`-based TLS model)

## Paging

The paging unit translates linear addresses to physical addresses using hierarchical **page tables**. The x86 architecture supports several paging modes:

### Paging Modes

| Mode | Levels | Page sizes | Physical address bits | CR4 flags |
|------|--------|------------|----------------------|-----------|
| **Standard 32-bit** | 2 (PGD + PTE) | 4 KB | 32 (4 GB) | — |
| **PSE (large pages)** | 1 (PGD only) | 4 MB | 32 (4 GB) | CR4.PSE=1 |
| **PAE** | 3 (PDPT + PGD + PTE) | 4 KB, 2 MB | 36 (64 GB) | CR4.PAE=1 |
| **x86-64** | 4 (PML4 + PDPT + PGD + PTE) | 4 KB, 2 MB, 1 GB | 48 virtual / 52 physical | — (always on in long mode) |

### Standard Two-Level Paging (32-bit, no PAE)

A 32-bit linear address is split:

```
31          22 21         12 11          0
+------------+------------+------------+
|  Directory |    Table   |   Offset   |
|  (10 bits) |  (10 bits) |  (12 bits) |
+------------+------------+------------+
```

- **CR3** register holds the physical address of the Page Directory (1024 entries, one per process).
- **Page Directory Entry (PDE)**: points to a Page Table (1024 entries).
- **Page Table Entry (PTE)**: points to a 4 KB page frame.

### PAE (Physical Address Extension)

PAE extends physical addressing to 36 bits (64 GB) and adds a third level:

```
31   30 29           21 20          12 11          0
+-----+--------------+-------------+-------------+
|PDPT |   Directory  |    Table    |    Offset   |
| (2) |   (9 bits)   |   (9 bits)  |  (12 bits)  |
+-----+--------------+-------------+-------------+
```

The PDPT (Page Directory Pointer Table) has 4 entries. Each PDE and PTE is 8 bytes (vs. 4 bytes in standard mode) to accommodate the wider physical address.

### Linux's Four-Level Page Table Model

To support all architectures uniformly, Linux defines a four-level page table model:

```
PGD  ->  PUD  ->  PMD  ->  PTE  ->  Page Frame
```

| Level | Name | Full name |
|-------|------|-----------|
| 1 | PGD | Page Global Directory |
| 2 | PUD | Page Upper Directory |
| 3 | PMD | Page Middle Directory |
| 4 | PTE | Page Table Entry |

On architectures with fewer hardware levels, the unused levels are **folded** — they contain exactly one entry and are optimized away at compile time:

- **Standard 32-bit x86**: PUD and PMD are folded (2-level hardware)
- **PAE x86**: PUD is folded (3-level hardware: PDPT serves as PGD, Page Directory as PMD)
- **x86-64**: all four levels are active

### Page Table Entry Flags

Each PTE contains a physical page frame address and flag bits:

| Bit | Name | Meaning |
|-----|------|---------|
| 0 | **Present** | Page is in physical memory (if 0, access causes Page Fault) |
| 1 | **Read/Write** | 0 = read-only, 1 = read-write. Used for [concept-copy-on-write](concept-copy-on-write.md) protection |
| 2 | **User/Supervisor** | 0 = only ring 0 can access, 1 = ring 3 can access |
| 3 | **PWT** | Page Write-Through (cache policy) |
| 4 | **PCD** | Page Cache Disable |
| 5 | **Accessed** | Set by hardware on any access (used by page replacement algorithms) |
| 6 | **Dirty** | Set by hardware on write (used to determine if page must be written back) |
| 7 | **PAT/PSE** | Page Attribute Table index (in PTE) or Page Size Extension (in PDE — if set, PDE maps a 4 MB page directly) |
| 8 | **Global** | TLB entry is not flushed on CR3 reload (used for kernel pages, requires CR4.PGE=1) |
| 9-11 | **Available** | Free for OS use. Linux uses these for swap entry encoding and other purposes |

## TLB Management

The **Translation Lookaside Buffer** caches recent linear-to-physical address translations. TLB misses cause an expensive page-table walk. Linux must flush TLB entries when page table mappings change.

### TLB Flush Operations

| Function | Scope | Implementation |
|----------|-------|----------------|
| `__flush_tlb()` | Flush entire TLB on current CPU | Reload CR3 (writes to CR3 flush all non-global TLB entries) |
| `__flush_tlb_global()` | Flush all TLB entries including global pages | Toggle CR4.PGE off and on |
| `__flush_tlb_single(addr)` | Flush one TLB entry on current CPU | `invlpg` instruction |
| `flush_tlb_mm(mm)` | Flush all entries for an address space | IPI to all CPUs running that `mm` |
| `flush_tlb_page(vma, addr)` | Flush one page in an address space | IPI + `invlpg` on affected CPUs |
| `flush_tlb_range(vma, start, end)` | Flush a range of pages | IPI + flush on affected CPUs |

### Lazy TLB Mode

When switching from a user process to a **kernel thread** (which has no user-space address space), the kernel does not flush the TLB. Instead, it enters **lazy TLB mode**: the kernel thread borrows the previous process's `mm_struct` via `active_mm`. Since kernel threads only access the kernel portion of the address space (which is the same across all page tables), the user-space TLB entries are harmless and will be flushed on the next switch to a real user process. This optimization avoids expensive TLB flushes during kernel thread execution.

## Kernel Address Space Layout (32-bit x86)

The 4 GB linear address space is split:

```
0x00000000 - 0xBFFFFFFF  (3 GB)  User space
0xC0000000 - 0xFFFFFFFF  (1 GB)  Kernel space
```

`PAGE_OFFSET` = 0xC0000000. The kernel's virtual address space is further divided:

| Region | Virtual address range | Size | Purpose |
|--------|----------------------|------|---------|
| **Direct mapping** | `PAGE_OFFSET` to `PAGE_OFFSET + 896 MB` | 896 MB | Linear (identity) mapping of physical RAM: `virt = phys + PAGE_OFFSET`. Contains all kernel code, data, and slab caches. Covers ZONE_DMA and ZONE_NORMAL. |
| **8 MB gap** | — | 8 MB | Guard region to catch wild kernel pointer dereferences |
| **vmalloc area** | `VMALLOC_START` to `VMALLOC_END` | ~120 MB | Non-contiguous virtual-to-physical mappings via `vmalloc()` |
| **Persistent kmap** | `PKMAP_BASE` | 4 MB (1024 pages) | Long-term mappings of high-memory pages via `kmap()` |
| **Fix-mapped addresses** | `FIXADDR_START` to `FIXADDR_TOP` | ~few pages | Compile-time constant virtual addresses for special purposes |
| **Temporary kmap** | Within fix-mapped | Per-CPU | Very short-term atomic mappings via `kmap_atomic()` |

## Memory Zones

Physical memory is divided into zones based on hardware constraints:

| Zone | Physical range (typical) | Purpose |
|------|-------------------------|---------|
| **ZONE_DMA** | 0 - 16 MB | Pages usable by legacy ISA DMA controllers (which can only address the first 16 MB) |
| **ZONE_NORMAL** | 16 MB - 896 MB | Directly mapped into kernel address space. Most kernel allocations come from here. |
| **ZONE_HIGHMEM** | > 896 MB | Not permanently mapped into kernel address space. Must be temporarily mapped via `kmap()` or `kmap_atomic()` before the kernel can access it. Used for user-space pages on machines with > 896 MB RAM. |

On 64-bit systems, `ZONE_HIGHMEM` is unnecessary because the kernel address space is large enough to map all physical memory directly.

## High Memory Mapping

Pages in `ZONE_HIGHMEM` are not part of the kernel's direct mapping. To access them, the kernel uses:

### kmap() / kunmap()

`kmap(page)` creates a **persistent mapping** of a high-memory page into the `PKMAP_BASE` region. The mapping persists until `kunmap()` is called. If all persistent kmap slots are in use, `kmap()` **sleeps** until one becomes available. Therefore `kmap()` cannot be used in interrupt context.

Internally, the kernel maintains a `pkmap_count[]` array tracking the usage of each slot, and a `page_address_htable` hash table for reverse lookups (given a `struct page`, find its kmap address).

### kmap_atomic() / kunmap_atomic()

`kmap_atomic(page, type)` creates a **temporary mapping** using fix-mapped addresses. It is faster than `kmap()` (no hash table lookup, no sleeping) but the mapping is **per-CPU and per-type** — it must be released quickly, cannot be used across schedule points, and is preemption-disabled for its duration. The `type` parameter (e.g., `KM_USER0`, `KM_IRQ0`) selects which per-CPU fix-mapped slot to use.

## Fix-Mapped Addresses

Fix-mapped addresses are compile-time constant virtual addresses that map to arbitrary physical addresses. They provide a way to create virtual-to-physical mappings without modifying the kernel page tables at runtime. Uses include:

- Local APIC registers
- I/O APIC registers
- `kmap_atomic()` slots
- Early-boot console

The virtual address for each fix-mapped index is computed as `FIXADDR_TOP - (index * PAGE_SIZE)`, growing downward. The actual physical mapping is set at boot time via `set_fixmap()`.
