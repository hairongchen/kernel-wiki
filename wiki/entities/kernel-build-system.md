# Kernel Build System (Kbuild)

Kbuild is the Linux kernel's build system, built on top of GNU Make. It compiles the kernel image, loadable modules, and device tree blobs from a configuration file (`.config`) generated by [Kconfig](kernel-configuration.md). Kbuild handles recursive directory traversal, dependency tracking, conditional compilation based on `CONFIG_` variables, and cross-compilation for different architectures.

## Build Prerequisites

Building the kernel requires a specific set of tools, each version-matched to the kernel release:

| Tool | Minimum Version | Purpose |
|------|----------------|---------|
| gcc | Varies by kernel | C compiler; the kernel uses many gcc-specific extensions |
| binutils (as, ld) | Varies by kernel | Assembler and linker |
| make | 3.79.1+ | Build orchestration |
| module-init-tools | 0.9.10+ | `depmod`, `modprobe`, `insmod` for module management |

Required tool versions are documented in `Documentation/Changes` in the kernel source tree. Using mismatched versions is a common source of build failures.

## Primary Build Targets

### Building the Kernel

```sh
make                  # Build everything: vmlinux, modules, and arch-default image
make -j4              # Parallel build with 4 jobs (use N = number_of_CPUs * 2)
make bzImage          # Build only the compressed kernel image (x86)
make modules          # Build only loadable modules
```

The default target builds:
1. **vmlinux** -- the uncompressed ELF kernel image (used for debugging with gdb)
2. **bzImage** (on x86) -- the compressed, bootable kernel image placed in `arch/x86/boot/bzImage`
3. All configured loadable modules (`.ko` files)

### Installing

```sh
make modules_install               # Install modules to /lib/modules/<version>/
make install                        # Install kernel image + run distribution install scripts
make INSTALL_MOD_PATH=/mnt/target modules_install   # Install to alternate root
```

`make modules_install` creates the directory `/lib/modules/<version>/` containing:
- `kernel/` -- module files organized by subsystem (drivers/, fs/, net/, etc.)
- `modules.dep` -- dependency graph generated by `depmod`
- `modules.alias` -- device-to-module mapping from `MODULE_DEVICE_TABLE()` / `MODULE_ALIAS()` declarations
- `modules.symbols` -- exported symbol-to-module mapping
- `modules.pcimap`, `modules.usbmap` -- device ID tables for hotplug matching

### Informational Targets

```sh
make kernelversion     # Print the kernel version string
make kernelrelease     # Print full release string including EXTRAVERSION and LOCALVERSION
make help              # List all available make targets
```

### Cleaning Targets

```sh
make clean             # Remove generated files but keep .config and enough for module builds
make mrproper          # Remove all generated files including .config
make distclean         # mrproper + remove editor backup files and patch leftovers
```

### Configuration Targets

See [kernel-configuration](kernel-configuration.md) for details.

```sh
make menuconfig        # Console-based configuration
make xconfig           # Qt-based graphical configuration
make oldconfig         # Update .config for new kernel version
make defconfig         # Default configuration for the architecture
```

### Documentation and Analysis Targets

```sh
make htmldocs / pdfdocs / mandocs    # Build kernel documentation
make tags / TAGS / cscope            # Generate editor tag files
make namespacecheck                  # Check for namespace conflicts
make headers_install                 # Install sanitized kernel headers for userspace
```

### Packaging Targets

```sh
make rpm-pkg           # Build an RPM package of the kernel
make deb-pkg           # Build a Debian package
make tar-pkg           # Build a tarball
make binrpm-pkg        # Build binary-only RPM
```

## The Kbuild Makefile System

### Kbuild Makefiles

Each directory in the kernel source tree contains a `Makefile` (or `Kbuild` file) that specifies what to build. The syntax ties directly to Kconfig variables:

```makefile
obj-y += foo.o              # Always compile foo.c into the kernel
obj-m += bar.o              # Compile bar.c as a loadable module
obj-$(CONFIG_BAZ) += baz.o  # Compile baz.c based on Kconfig setting:
                            #   CONFIG_BAZ=y -> built-in
                            #   CONFIG_BAZ=m -> module
                            #   CONFIG_BAZ not set -> not compiled
```

Multi-file modules are specified as:

```makefile
obj-$(CONFIG_MY_DRIVER) += my_driver.o
my_driver-objs := my_driver_core.o my_driver_io.o my_driver_dma.o
```

### The Top-Level Makefile

The top-level `Makefile` contains the kernel version information:

```makefile
VERSION = 2
PATCHLEVEL = 6
SUBLEVEL = 19
EXTRAVERSION =
NAME = Crazed Snow-Weasel
```

The `EXTRAVERSION` field can be modified to append a custom string (e.g., `-custom` or `-debug`) to the kernel version. This is useful for distinguishing custom-built kernels. The `CONFIG_LOCALVERSION` Kconfig option provides an equivalent mechanism via the configuration system.

## Cross-Compilation

Kbuild supports building the kernel for a different architecture than the host machine:

```sh
make ARCH=arm CROSS_COMPILE=arm-linux-gnueabi-
```

| Variable | Purpose |
|----------|---------|
| `ARCH` | Target architecture (arm, mips, powerpc, x86, etc.) |
| `CROSS_COMPILE` | Prefix for the cross-toolchain binaries (gcc becomes `arm-linux-gnueabi-gcc`) |

These variables can be set as environment variables or passed on the make command line. When `ARCH` differs from the host architecture, Kbuild uses the architecture-specific configuration, assembly files, and linker scripts from `arch/<ARCH>/`.

## Out-of-Tree Builds

Kbuild supports building with the source and output in separate directories, keeping the source tree pristine:

```sh
make O=/path/to/output
```

This creates all generated files (`.o`, `.ko`, `.config`, `vmlinux`, `bzImage`) under the specified output directory. The source directory remains unmodified. This is useful for:
- Building multiple configurations from a single source tree
- Building on a read-only source tree (e.g., NFS-mounted)
- Keeping generated files out of version control

## Build Acceleration

### Parallel Builds

```sh
make -j$(nproc)           # Use all available CPUs
make -j8                  # Explicit 8 parallel jobs
```

A good rule of thumb is `N = number_of_CPUs * 2` to keep CPUs busy during I/O waits. On large SMP machines, parallel builds provide near-linear speedup.

### ccache (Compiler Cache)

```sh
make CC="ccache gcc"
```

ccache hashes the preprocessed source and compiler flags. If the hash matches a previous compilation, the cached object file is returned without invoking the compiler. Dramatically speeds up rebuilds after minor changes or `make clean`.

### distcc (Distributed Compilation)

```sh
make CC="distcc gcc"
```

distcc distributes compilation jobs across multiple machines on the network. Combined with `-jN` (where N exceeds local CPU count), this leverages available build capacity across a cluster. Preprocessing and linking still occur locally; only compilation is distributed.

## Partial Builds

Build only files in a specific subdirectory:

```sh
make drivers/usb/         # Build all files in drivers/usb/
make M=drivers/usb/       # Build only external modules in that directory
```

This is useful during driver development to avoid rebuilding the entire kernel.

## Verifying Module Configuration

To find the Kconfig option that builds a particular module:

```sh
# In the kernel source, search for the .o target in Makefiles
grep -r "my_driver.o" drivers/
# Output: drivers/net/Makefile:obj-$(CONFIG_MY_DRIVER) += my_driver.o
# -> Enable CONFIG_MY_DRIVER in menuconfig
```
