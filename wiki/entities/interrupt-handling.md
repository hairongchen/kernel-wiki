# Interrupt Handling

Interrupts and exceptions are the fundamental mechanisms by which the CPU responds to asynchronous hardware events and synchronous error conditions. The Linux kernel divides interrupt processing into a fast **top half** (runs immediately with interrupts potentially disabled) and a deferred **bottom half** (runs later in a more relaxed context), balancing response latency against the need to minimize time spent with interrupts disabled.

## The Interrupt Descriptor Table (IDT)

The x86 architecture uses a 256-entry **Interrupt Descriptor Table** to map each interrupt or exception vector (0-255) to its handler. The IDT is populated at boot time by `set_intr_gate()`, `set_trap_gate()`, and `set_system_gate()`. Each entry is an 8-byte descriptor containing the handler's segment selector and offset, plus control bits.

### Gate Types

| Gate type | IF cleared? | DPL | Purpose |
|-----------|-------------|-----|---------|
| **Interrupt gate** | Yes (interrupts disabled on entry) | 0 | Hardware IRQs — prevents nested handling of the same IRQ line |
| **Trap gate** | No (interrupts remain enabled) | 0 | Most exceptions — allows other interrupts during exception handling |
| **System gate** | No | 3 | Vectors accessible from user mode (e.g., `int $0x80` for system calls, `int3` for breakpoints) |

Vectors 0-31 are reserved by Intel for exceptions. Vectors 32-255 are available for hardware interrupts and software-defined purposes.

## Exception Handling

An **exception** is a synchronous event generated by the CPU when it detects an anomalous condition during instruction execution. Exceptions fall into three classes:

| Class | Behavior | Example |
|-------|----------|---------|
| **Fault** | Saved `EIP` points to the faulting instruction; the handler can fix the condition and **restart** it | Page Fault (#14), General Protection Fault (#13) |
| **Trap** | Saved `EIP` points to the instruction **after** the one that caused the trap; execution continues at the next instruction | Breakpoint (#3), Overflow (#4) |
| **Abort** | Severe, unrecoverable error; the process (or the system) cannot continue | Machine Check (#18), Double Fault (#8) |

### Key Exception Vectors

| Vector | Name | Type | Handler |
|--------|------|------|---------|
| 0 | Divide Error | Fault | `divide_error()` |
| 1 | Debug | Trap/Fault | `debug()` |
| 3 | Breakpoint | Trap | `int3()` — system gate (DPL=3) |
| 6 | Invalid Opcode | Fault | `invalid_op()` |
| 7 | Device Not Available | Fault | `device_not_available()` — triggers lazy FPU restore |
| 12 | Stack Segment Fault | Fault | `stack_segment()` |
| 13 | General Protection | Fault | `general_protection()` |
| 14 | Page Fault | Fault | `page_fault()` -> `do_page_fault()` — the most performance-critical exception handler, central to [concept-copy-on-write](../concepts/concept-copy-on-write.md) and demand paging |

Most exception handlers send a signal to the current process (e.g., `SIGSEGV` for invalid memory access, `SIGFPE` for arithmetic errors). The Page Fault handler is unique in that it often resolves the fault transparently by allocating pages or performing copy-on-write.

## Interrupt Controllers: PIC and APIC

### 8259A PIC

Uniprocessor systems use a pair of cascaded **8259A Programmable Interrupt Controllers**, providing 15 usable IRQ lines (IRQ 0-15, with IRQ 2 used for cascading). The PIC is programmed via I/O ports. Each IRQ can be individually masked. The PIC delivers interrupts to a single CPU and requires explicit end-of-interrupt (EOI) acknowledgment.

### APIC Architecture (SMP)

SMP systems use the **Advanced Programmable Interrupt Controller** architecture:

- **Local APIC**: One per CPU, integrated into the processor. Handles local interrupt sources (timer, performance counters, thermal, error, and inter-processor interrupts / IPIs). Each local APIC has a 32-bit ID used for interrupt routing.
- **I/O APIC**: Typically one per system (on the chipset). Receives external hardware interrupts and routes them to one or more local APICs based on a 24-entry redirection table. Each entry specifies the destination CPU(s) and delivery mode (fixed, lowest-priority, NMI, etc.).

The APIC allows **symmetric interrupt distribution** across CPUs, which is essential for SMP scalability. The kernel programs the I/O APIC redirection table to balance IRQs across processors. Inter-processor interrupts (IPIs) are used for TLB shootdowns, reschedule notifications, and function-call broadcasts.

## Interrupt Handling Path

When an external hardware interrupt arrives:

```
Hardware IRQ
  -> IDT lookup (vector 32-255)
    -> common_interrupt (assembly stub)
      -> do_IRQ(regs)
        -> irq_enter()    /* increment preempt_count, note interrupt context */
        -> __do_IRQ(irq)
          -> ack interrupt (via irq_desc[irq].handler->ack)
          -> walk irqaction chain, call each handler
          -> end interrupt (via irq_desc[irq].handler->end)
        -> irq_exit()     /* decrement preempt_count, run softirqs if pending */
      -> ret_from_intr
```

### The irq_desc[] Array

The kernel maintains an array of `irq_desc_t` structures, one per IRQ line. Each entry contains:

| Field | Purpose |
|-------|---------|
| `handler` | Pointer to the `hw_irq_controller` for this IRQ (8259A PIC, I/O APIC, etc.) — provides `ack()`, `end()`, `enable()`, `disable()` operations |
| `action` | Linked list of `irqaction` structures, one per registered handler for this IRQ |
| `status` | Flags: `IRQ_DISABLED`, `IRQ_INPROGRESS`, `IRQ_PENDING`, etc. |
| `depth` | Nesting depth for `disable_irq()` / `enable_irq()` |
| `lock` | Spinlock protecting this descriptor |

### Registering and Freeing Handlers

Drivers register interrupt handlers with `request_irq()`:

```c
int request_irq(unsigned int irq,
                irqreturn_t (*handler)(int, void *, struct pt_regs *),
                unsigned long flags,
                const char *devname,
                void *dev_id);
```

Key flags:
- `SA_INTERRUPT` — run with all interrupts disabled on this CPU (fast handler)
- `SA_SHIRQ` — shared IRQ line (multiple devices on one vector); `dev_id` distinguishes them
- `SA_SAMPLE_RANDOM` — use interrupt timing to feed the entropy pool

`free_irq()` removes a handler. For shared IRQs, the `dev_id` identifies which handler to remove.

### Return Path: ret_from_intr / ret_from_exception

The assembly return paths `ret_from_intr` and `ret_from_exception` perform critical checks before returning to the interrupted context:

1. If returning to **kernel mode**: check `preempt_count` and `TIF_NEED_RESCHED` — if preemption is enabled and rescheduling is needed, call `schedule()` (kernel preemption).
2. If returning to **user mode**: check `TIF_NEED_RESCHED` and call `schedule()` if set. Check `TIF_SIGPENDING` and deliver pending signals. These are the main voluntary preemption points.

## Top Half vs. Bottom Half

The **top half** is the interrupt handler registered via `request_irq()`. It runs in interrupt context (no `current` process, cannot sleep, cannot access user space) with at least the current IRQ line disabled. Top halves should do the minimum work necessary: acknowledge the hardware, copy data out of device buffers, and schedule deferred work.

The **bottom half** runs deferred work outside the critical interrupt window, with interrupts re-enabled. Linux 2.6 provides three bottom-half mechanisms.

## Deferred Work Mechanism 1: Softirqs

Softirqs are the lowest-level, highest-priority deferred execution mechanism. They are **statically allocated** at compile time (no dynamic registration). In 2.6.11, there are six softirq types:

| Index | Name | Purpose |
|-------|------|---------|
| 0 | `HI_SOFTIRQ` | High-priority tasklets |
| 1 | `TIMER_SOFTIRQ` | Timer callbacks |
| 2 | `NET_TX_SOFTIRQ` | Network transmit |
| 3 | `NET_RX_SOFTIRQ` | Network receive |
| 4 | `SCSI_SOFTIRQ` | SCSI post-processing |
| 5 | `TASKLET_SOFTIRQ` | Normal-priority tasklets |

### Softirq Properties

- **Per-CPU**: each CPU has its own set of pending softirq flags (a bitmask in `irq_cpustat_t`). A softirq raised on CPU 0 runs on CPU 0.
- **Reentrant**: the same softirq can run simultaneously on different CPUs. Softirq handlers must use their own locking for shared data.
- **Non-preemptible**: softirqs run with hardware interrupts enabled but with preemption disabled (`in_softirq()` returns true).

Softirqs are raised with `raise_softirq(nr)` (typically from a top-half handler) and are processed by `do_softirq()` at specific points:
1. After returning from a hardware interrupt (`irq_exit()`)
2. In the `ksoftirqd` kernel thread
3. In code that explicitly calls `do_softirq()`

### ksoftirqd Throttling

If softirqs keep re-raising themselves (common under heavy network load), `do_softirq()` does not loop indefinitely — it processes them in a single pass, then wakes the per-CPU `ksoftirqd/N` kernel thread to handle any remaining work. The ksoftirqd thread runs at the lowest nice level (nice 19), ensuring that softirq processing does not starve user-space processes under load. This is a deliberate trade-off: under light load softirqs run immediately at interrupt-return time (low latency); under heavy load they are deferred to ksoftirqd (fairness).

## Deferred Work Mechanism 2: Tasklets

Tasklets are built on top of softirqs (`HI_SOFTIRQ` and `TASKLET_SOFTIRQ`) but provide a simpler interface for drivers:

- **Dynamic**: tasklets can be created and destroyed at runtime.
- **Serialized per-tasklet**: a given tasklet is guaranteed to run on only one CPU at a time (the same tasklet cannot be re-entered concurrently), eliminating the need for locking within the tasklet handler.
- **Not serialized across different tasklets**: different tasklets can run in parallel on different CPUs.

A tasklet is represented by `tasklet_struct`:

```c
struct tasklet_struct {
    struct tasklet_struct *next;
    unsigned long state;    /* TASKLET_STATE_SCHED, TASKLET_STATE_RUN */
    atomic_t count;         /* 0 = enabled, nonzero = disabled */
    void (*func)(unsigned long);
    unsigned long data;
};
```

Tasklets are scheduled with `tasklet_schedule()` (normal priority) or `tasklet_hi_schedule()` (high priority). Each CPU maintains per-CPU lists of pending tasklets. When the softirq fires, it walks the list and calls each tasklet's function, checking the `TASKLET_STATE_RUN` flag to ensure single-CPU execution.

## Deferred Work Mechanism 3: Work Queues

Work queues allow deferred work to run in **process context**, meaning the handler can **sleep** (unlike softirqs and tasklets, which run in interrupt/softirq context). This makes work queues suitable for tasks that need to allocate memory with `GFP_KERNEL`, acquire semaphores, or perform blocking I/O.

### Implementation

The kernel creates a default set of worker threads called `events/0`, `events/1`, ..., `events/N` (one per CPU). Each thread processes a queue of `work_struct` entries:

```c
struct work_struct {
    unsigned long pending;
    struct list_head entry;
    void (*func)(void *);
    void *data;
    void *wq_data;
    struct timer_list timer;
};
```

Drivers submit work with `schedule_work()` (to the default `events/n` queue) or `queue_work()` (to a custom work queue). The worker thread wakes up, dequeues the `work_struct`, and calls `func(data)`.

Drivers can create custom work queues with `create_workqueue()`, which spawns a dedicated set of per-CPU kernel threads. This is useful when the work is long-running or might block the shared `events/n` threads.

### Comparison of Bottom-Half Mechanisms

| Property | Softirq | Tasklet | Work Queue |
|----------|---------|---------|------------|
| Context | Softirq (atomic) | Softirq (atomic) | Process (can sleep) |
| Concurrency | Same softirq can run on multiple CPUs | Per-tasklet serialization | Per-CPU worker thread |
| Allocation | Static (compile-time) | Dynamic | Dynamic |
| Use case | High-throughput subsystems (networking, block I/O) | Most driver bottom halves | Work that needs to sleep |
| Reentrancy | Must handle its own locking | Automatically serialized | Process context rules apply |

## genirq Framework (Linux 2.6.24)

Linux 2.6.24 introduced the **genirq (generic IRQ)** framework, replacing the previous architecture-specific interrupt handling code. genirq cleanly separates two concerns that were previously intermingled:

### IRQ Chip Abstraction

**`struct irq_chip`** encapsulates the hardware operations for an interrupt controller:

| Method | Purpose |
|--------|---------|
| `ack()` | Acknowledge the interrupt |
| `mask()` / `unmask()` | Disable / enable the IRQ line |
| `mask_ack()` | Combined mask + acknowledge |
| `eoi()` | End-of-interrupt signal |
| `set_type()` | Configure edge vs. level triggering |
| `set_affinity()` | Set CPU affinity for the IRQ |

This replaces the older `hw_irq_controller` structure with a cleaner, more portable interface.

### Flow Handlers

**Flow handlers** encapsulate the interrupt flow logic — the sequence of ack, mask, handler invocation, and unmask operations that differs between edge-triggered and level-triggered interrupts:

- `handle_level_irq()` — mask → ack → handle → unmask (level-triggered)
- `handle_edge_irq()` — ack → handle, with re-check for edges received during handling
- `handle_fasteoi_irq()` — handle → eoi (modern APIC with automatic masking)
- `handle_simple_irq()` — handle only (no ack/mask, for software-generated IRQs)

Each IRQ line's `irq_desc` now has a `handle_irq` function pointer set to the appropriate flow handler, and a `chip` pointer to the `irq_chip`. The flow handler calls chip methods as needed, then iterates the `irqaction` chain.

### IRQ Affinity

The genirq framework exposes `/proc/irq/<N>/smp_affinity` for per-IRQ CPU affinity control. Writing a CPU bitmask causes `irq_chip->set_affinity()` to reprogram the I/O APIC redirection entry, steering the IRQ to specific CPUs.
